* 测试
* benchmark
* rollout / rollback

它不擅长的是高度模糊、无法验收、没有真相回流的任务。

**可 contract 化工作的自治极限。**

## 2. 可验证边界

它能走多远，取决于结果能不能被验证。

* 测试能否自动跑
* benchmark 能否自动比
* receipt 能否准确记录
* reconciliation 能否形成 verdict

越可验证，Hermit 越强。
越不可验证，它越只能做辅助而不是自治。

Hermit 必须真的能作用于环境：

* 查状态
* 发起审批
* 更新任务状态

没有操作面，它就只是一个会写计划的脑子。
所以它的上限也取决于：
**环境是否被“工具化 / API 化 / contract 化”。**

---

## 4. 治理可承受边界

这类系统最大的天花板，不是智力，而是治理成本。

如果：

* contract 漂移严重
* verification 跟不上
* memory 污染
* 并发互踩
* 状态读模型不清晰

那它很快就会从“task OS”退化成“会制造噪音的 agent swarm”。

所以真正的上限是：

**在多大复杂度下，控制面还能压住执行面。**

如果这三个东西有了，你就只需要给大方向，系统自己能长出工作。

---

# 七、你可以给自己定一个很实用的衡量指标

## Prompt Leverage Ratio

我建议你真的用这个指标。

```text
Prompt Leverage Ratio = 派生出的有效 attempts 数 / 人类输入 prompt 数
```

例如：

* 你发 1 个 prompt
* 系统最后执行了 18 个有效 attempts
* 其中 12 个被验证通过，3 个 follow-up，3 个被拒绝

那 leverage 已经很高了。

你后面追求的，不该是：

* 每天发多少 prompt

而是：

* **每个 prompt 能驱动多少有效闭环**

# 八、所以“没有那么多 prompt”其实反而说明你方向是对的

因为真正成熟的任务系统，本来就不该依赖人类持续喂 microscopic prompts。

它应该更像这样：

### 你给：

* 北极星目标
* 阶段性计划
* 风险预算
* 决策边界

### Hermit 自己长出：

* task graph
* follow-ups
* benchmark runs
* retries
* review loops
* nightly backlog work

如果它做不到这点，那你现在焦虑的不是“我没那么多 prompt”，而是：

**Hermit 还没形成自增长工作流。**

这才是应该解决的根因。

# 九、一句话结论

**如果你派发不出去这么多 prompt，不是你的问题，而是系统还没把“高层意图”放大成“持续任务流”。**

你真正要做的，不是逼自己发更多 prompt，
而是让 Hermit 具备：

* **从 Program 自动拆任务**
* **从失败自动生 follow-up**
* **从空闲自动挑 backlog**
* **从证据自动续跑下一轮**
