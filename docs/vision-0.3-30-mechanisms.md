# Hermit 0.3 的 50 个机制

> 不讲实现细节。每条只说"它能做什么"和"为什么这很重要"。

---

## 治理：每个动作都有边界

**1. 没有 Agent，只有 Task**
系统里没有 Agent 类、没有 Agent 生命周期。你提交的是任务，系统管理的是执行。任务可以暂停、恢复、重试、回滚、跨天存活。对话只是表达意图的方式，不是工作的容器。

**2. 每个动作过 14 步守卫链**
从委派范围检查到文件系统规则、Shell 命令规则、信号风险升级、信任评分建议……一共 14 步顺序评估。多条规则的裁决取最严。如果走完没有任何规则明确放行，默认是**需要审批**。

**3. 权限是一张 300 秒过期的精确许可证**
不是"给你读写权限"。精确到：你只能写这个路径、只能访问这些主机、Shell 命令必须完全匹配这一条。300 秒后自动失效，必须重新走一遍评估。

**4. 同一时刻只有一个任务能改同一个目录**
如果有人占着，你的请求进队列等——不是被拒绝。释放时自动把队列里下一个请求放出来。两个任务永远不会同时修改同一个文件。

**5. 高风险决策要经过正式辩论**
多个候选方案各自提交提案（含成本/风险/收益估算），评审者发出带证据的分级质疑，仲裁者产出带置信度的裁决。辩论过程打包存档，可审计可重放。仲裁置信度不够高时自动升级给人类。

**6. 信任是一个持续计算的分数，不是一个开关**
基于成功率、回滚率、和解置信度算出 0–1 的信任分。但危险动作类不管分数多高都不会降级。分数只是建议——最终决定权永远在守卫链。

**7. 治理会随时间变轻**
系统从成功的执行中学习合同模板。同类操作被成功执行 3 次以上、成功率超过 85%，系统会建议简化后续审批。第一次最重，第十次几乎透明。遇到新类型操作——治理自动回到全强度。

**8. sudo 和 curl|sh 是硬编码的死线**
不管你的信任分多高、policy profile 多宽松、模板匹配多完美——`sudo` 和 `curl | sh` 永远被拒绝。有些事情系统永远不会帮你做。

## 执行：阻塞不等于停机

**9. 19 个显式状态追踪每个治理阶段**
dispatching → contracting → preflighting → policy_pending → awaiting_approval → observing → reconciling → receipt_pending……每个阶段持久化。任何位置崩溃都能恢复。你可以在事后精确重放一个动作经历了什么。

**10. 卡住时自动换一件事做**
任务等审批时，系统自动泊车，按优先级选下一个能做的任务继续推进。审批通过后重新评估优先级——恢复的任务不一定排第一了。**永远在推进当前能推进的最高优先级工作。**

**11. 不是所有并行都需要"全部成功"**
四种汇合策略：全部成功、任一成功、多数通过、尽力而为。竞争式执行用"任一成功"——第一个过 benchmark 的方案胜出。信息收集用"尽力而为"——能拿多少拿多少。

**12. 长时间任务不会占住 worker**
CI pipeline、webhook 回调这类慢任务，系统创建观察票、释放 worker slot，后台轮询进度。检测到完成信号时自动恢复。观察票跨重启存活。

**13. 崩溃恢复是分叉，不是回滚**
恢复时 fork 一条新任务分支，上游步骤标记跳过，下游重新就绪。原始事件流完全不动。不是抹掉历史重来，是从断点长出新分支。

**14. 重启时区分"可能已执行"和"肯定没执行"**
有权限令牌的 → 动作可能已经执行了 → 阻塞等人工确认。没有令牌的 → 动作从未被授权 → 安全重入。**绝不悄悄假设"大概没执行吧"。**

**15. 每个命令有软硬两道截止线**
软截止触发警告，硬截止直接终止进程。跑飞的工具不会耗尽整个执行预算。像安全气囊——希望永远不用，但必须有。

**16. 僵尸任务 7 天后自动死亡**
卡在 planning_ready / paused / blocked / needs_attention 超过 7 天的任务自动标记失败。暂停的任务走 cancelled（不是 failed），因为它没做错什么——只是等太久了。

## 证据与审计：AI 说"做完了"不算数

**17. Receipt 是签名的结构化记录，不是 log line**
每个有后果的动作产出 HMAC-SHA256 签名的 Receipt，关联到 task、step、policy 决定、审批、权限令牌、工作区租约、回滚策略。改一个字节签名就断。

**18. 三级证据导出：从摘要到 Merkle 证明**
Summary（几 KB）→ Standard（几十 KB，加全部治理记录）→ Full（MB 级，含 Merkle inclusion proof）。按需选择。你可以把 proof bundle 交给第三方独立验证。

**19. 事件日志是密码学链接的**
每个事件的 hash 包含前一个事件的 hash。篡改、插入、删除任何一个事件 → 链断裂。46 张表，12 个零耦合模块组合。这不是日志——是账本。

**20. 和解有 9 种精确结论**
satisfied / violated / drifted / rolled_back / partial / ambiguous……每种结论映射一个推荐下一步和置信度增量（成功 +0.2，违反 -0.3）。和解不是"记录结果"，是**告诉系统接下来该怎么办**。

**21. 回滚知道执行前的精确状态**
每个可回滚的动作记录执行前的精确快照（文件内容 + hash、git 状态、记忆 ID）。回滚前校验快照 hash——防止在已被篡改的状态上回滚。不是"撤销上一步"，是"精确恢复到那一刻"。

**22. 执行前会评估"证据够不够"**
不是 pass/fail，而是加权分数。状态快照最重要，其次是 policy 结果、上下文、动作请求。系统还会告诉你"这些证据有多容易过时"。

## 记忆与认知：不只是"记住了"

**23. 每晚五遍记忆整固——系统的"睡眠"**
去重 → 强化高频引用 → 衰减过期 → 从聚类中合成更高阶洞察 → 把持续失败的记忆反转成反面教训。这不是垃圾回收，是**认知整理**。

**24. 失败会被反转成最高优先级的警告**
当一条记忆持续导致失败，系统把它反转成 PITFALL 警告，置信度放大 4 倍。在工作记忆中，"不要做什么"永远排在"应该做什么"前面。

**25. 记忆有四种鲜度，过期不是删除**
FRESH → AGING → STALE → EXPIRED。过期进入隔离区。如果新证据出现，隔离记忆可以复活。用户偏好 365 天过期，临时事实 1 天过期。**被隔离不等于被否定——只是暂时存疑。**

**26. 系统会从实例中自动提炼规律**
3 条以上相关记忆聚在一起 → 自动合成"X 总是导致 Y"、"这个模式反复出现"、"这两条矛盾信念应该这样调和"。可以区分"直接观察"和"推理得出"。

**27. 记忆晋升只接受被验证过的证据**
从临时信念晋升为持久记忆，必须引用经过完整治理流水线的和解记录。模型说"我觉得是这样"不算——必须有执行证据支撑。

**28. 每条记忆的因果链可追溯**
任务失败 → 追溯这个决策参考了哪些记忆 → 检查那些记忆的历史可靠性。反复导致失败的"坏顾问"记忆会被标记出来。

**29. 上下文是 4 路检索融合的结果**
Token 重叠、语义嵌入、知识图谱、时间衰减——4 条独立路径各自排名，融合后给出最终结果。每条被选中和被排除的记忆都标记了原因。**你可以审查：模型看到了什么，没看到什么，为什么。**

**30. 截图也是可搜索的记忆**
你发给 Hermit 的截图会被 SHA-256 索引、OCR 提取文字、打上内容标签。之后可以像搜索文字记忆一样搜索图片。截图变成了证据链的一部分。

**31. 系统能从经验中提取"操作手册"**
检测到 episodic 记忆中的 how-to 模式（"先做 X 再做 Y 最后做 Z"），自动提取为可复用的 procedure。有成功/失败计数器——失败多了的 procedure 会被降级。

## 自主性：你不在的时候它也在干活

**32. 系统自己监控代码健康**
后台守护进程每小时跑一次 lint、测试、覆盖率、安全扫描。发现问题自动发射证据信号。你没让它查——它自己查。

**33. 失败自动长出修复任务**
每次执行后扫描结果，用规则匹配问题模式。匹配到 → 自动生成带证据引用的修复任务。每条规则有冷却期防止刷屏。**失败不只是被记录——它会繁殖出新工作。**

**34. 自己改自己的代码必须过三道门**
在隔离 worktree 中执行修改 → 冒烟测试 → 受影响测试 → 完整检查。第一道不过就停。全过了 + 和解通过才合并回主分支。worktree 被丢弃时主分支没有任何影响。

**35. 你可以在执行中途转向**
发出转向指令（调整范围 / 策略 / 约束 / 优先级），系统必须明确回应——接受、应用、或拒绝。不是取消重来，是实时校正航向。新指令和旧指令冲突时，旧指令被标记为被取代。

**36. 你睡着了它会给你准备早报**
Overnight 模式持续推进低风险任务，你醒来看到的是一份结构化的晨间报告：完成了什么、失败了什么、哪些在等你审批、治理指标有没有异常。不是聊天记录——是**管理层简报**。

**37. 检测到的问题会自动变成改进规划**
证据信号（代码警告、测试失败、风险标记）自动进入 spec backlog，变成未来迭代的候选改进项。系统不只是报告问题——它把问题转化成可执行的改进计划。

**38. 每次迭代结束都会提取经验教训**
benchmark 结果和 proof chain 被分析，提取成功模式、回滚模式和错误模式，存入 lesson pack。下一轮迭代启动时，这些教训会被纳入上下文。**系统的每一次失败都在为下一次成功做准备。**

## 协作：不止你一个人能给它派活

**39. 你可以从飞书 / Slack / Telegram 直接派任务**
不需要打开终端。发一条消息就是提交一个任务。系统自动分类意图（新任务 / 查进度 / 控制命令），路由到正确的处理流程。

**40. 其他 AI 系统也可以给它派活**
通过 MCP 协议，Claude Code、Cursor、任何 MCP 客户端都可以向 Hermit 提交任务、等待完成、导出证据。Hermit 变成了一个**被治理的执行后端**——你的 AI 助手的 AI 下属。

**41. 外部系统可以通过 Webhook 触发任务**
HTTP webhook 带签名验证。GitHub push、CI 完成、监控报警——任何能发 HTTP 请求的系统都可以触发 Hermit 的治理流水线。

**42. 定时任务像 cron 一样精确**
后台调度器支持 cron 表达式，精确到分钟。启动时自动补跑错过的任务。执行历史完整记录。你可以让 Hermit 每天凌晨自动跑一遍代码健康检查。

**43. 消息到达时系统自动判断"这属于哪个任务"**
多信号评分：显式 task ID → 对话焦点 → 文件路径匹配 → 主题相似度。分数高且领先明显 → 自动归入。分数接近 → 主动问你"这条消息是说哪个任务？"。**不确定的时候不会猜。**

## 基础设施：看不见但关键

**44. 8 种 worker 角色，每种有独立的并发上限**
planner / executor / verifier / benchmarker / tester / researcher / reconciler / spec——每种角色有独立的 slot 数量限制。不会出现"10 个 planner 在跑但没有 executor"的失衡。同一个 workspace 最多 1 个 executor，同一个 module 最多 2 个。

**45. 系统健康有一个实时分数**
任务吞吐量、审批通过率、平均审批延迟、回滚率、证据充分性、各动作类别分布——聚合成一份实时健康报告。你随时可以问"系统状态怎么样"，得到的不是 log，是指标。

**46. 中英双语，系统级支持**
所有面向用户的输出（错误信息、审批描述、状态报告）都有 en-US 和 zh-CN 两套。不是翻译后贴上去的——是从 locale 文件里取的结构化文本。意图识别也同时匹配中英文关键词。

**47. macOS 菜单栏伴侣应用**
一个常驻菜单栏的小应用，显示 Hermit 服务状态、快速操作入口。和 `hermit serve` 配合使用，不需要一直开着终端窗口。

**48. 热重载不杀后台进程**
重新加载配置和插件时，调度器、webhook 服务器、巡逻引擎保持运行不中断。你可以改插件代码然后 reload，不需要重启整个服务。

**49. 多模型随时切换**
在 config.toml 里定义 profile（Claude Sonnet / Claude Opus / Codex / 自定义），运行时随时切换。不同任务可以用不同模型——规划用 Opus，执行用 Sonnet，不浪费预算。

**50. 内容寻址的制品存储**
制品按 SHA-256 hash 存储。同样的内容永远只存一份。路径按 hash 前缀分层（像 git objects）。写入是原子的（临时文件 → 重命名），读写中途崩溃不会产出损坏文件。
