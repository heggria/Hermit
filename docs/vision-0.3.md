# Hermit 0.3：没有 Agent 的 Agent 系统

## 一句话

**在 Hermit 0.3 里，已经没有"Agent"的概念了。作为人类，你做的事情只有一件：向 Hermit 提交一个可执行的任务。**

---

## 问题：我们到底在和谁说话？

过去两年，整个行业都在讨论"AI Agent"。你给它一个 prompt，它思考，它调工具，它给你回答。你和它之间的关系是：**对话**。

但你有没有想过一个问题——

当你让一个 agent 修改 20 个文件、跑测试、提 PR、处理 review 意见的时候，你到底是在"对话"，还是在"派活"？

答案显然是后者。你不是在和一个聊天伙伴说话。你是在向一个**执行系统**提交工作。

Session 结束了，工作应该还在。审批卡住了，其他任务应该继续跑。你睡着了，低风险的工作不应该停。你第二天醒来，应该看到的不是一个等你发消息的聊天框，而是一份**进展报告**。

这不是一个 agent 该有的样子。这是一个**操作系统**该有的样子。

---

## Hermit 0.3 的答案：Task，不是 Agent

Hermit 0.3 做了一个根本性的选择：**把 Agent 这个概念彻底从架构中移除。**

没有 Agent 类。没有 Agent 生命周期。没有"创建一个 Agent"这样的操作。

取而代之的，是一个完整的任务操作系统：

```
你（人类）
    ↓ 提交目标
Hermit（Task OS）
    ↓ 编译成
Program → Team → Task → Step → StepAttempt
    ↓ 每一步都经过
Policy → Approval → CapabilityGrant → WorkspaceLease → Execution → Receipt
    ↓ 产出
Artifacts + Receipts + Proofs + Reconciliation → Durable Learning
```

你提交的是**任务**，不是消息。
系统管理的是**执行**，不是对话。
留下的是**证据链**，不是聊天记录。

---

## 为什么要杀掉 Agent？

### 1. Agent 是一个模糊的隐喻

"Agent"这个词暗示了一个拟人化的实体——它有意志，有判断，有自主性。这个隐喻在早期很有用，它帮助人们理解"AI 可以主动做事"。但当系统变得复杂时，这个隐喻开始伤害你。

因为当你说"我有一个 Agent"的时候，你其实不知道：

- 它正在做什么？（没有 task identity）
- 它被允许做什么？（没有 capability scope）
- 它做过什么？（没有 receipts）
- 做错了能撤回吗？（没有 rollback）
- 它的"记忆"靠不靠谱？（没有 evidence binding）

"Agent"给了你一种控制感的幻觉。但幻觉不是控制。

### 2. Session 是错误的执行原语

几乎所有 Agent 框架的底层原语都是 **session**——一段对话。工作在 session 里发生，session 结束，工作就结束了。

但真实的工作不是这样的：

- 工作需要**审批**——你不能在 session 里等审批
- 工作需要**恢复**——session 崩了工作不能丢
- 工作需要**并行**——一个 session 跑不了十件事
- 工作需要**审计**——"聊天记录"不是审计材料

Session 适合问答。不适合执行。

Hermit 的选择是：**Task 才是正确的执行原语。**

Task 有 identity。Task 有 lifecycle。Task 可以暂停、恢复、重试、回滚。Task 可以在你不在的时候继续推进。Task 在一年之后依然可以被检查和验证。

### 3. 权威不能是环境的

在传统 Agent 框架里，当模型调用一个工具的时候，它继承的是**进程级别的权限**。你的文件系统、你的 shell、你的 API key——全部暴露在模型的执行路径上。

这就像 1970 年代的操作系统：所有程序共享同一块内存，任何一个程序的 bug 都可以搞崩整个系统。

现代操作系统解决这个问题靠的是：进程隔离、权限控制、系统调用。

Hermit 做的是完全一样的事：

| 操作系统概念 | Hermit 对应 |
|---|---|
| 进程 | Task |
| 系统调用 | Governed Action |
| 文件权限 | CapabilityGrant |
| 进程隔离 | WorkspaceLease |
| 审计日志 | Receipt + Proof |
| 内核模块 | Plugin |
| `/proc` 文件系统 | Kernel Ledger |
| 进程组 | Program |
| 用户账户 | Team + Role |
| CPU 调度器 | Worker Pool Manager |

这不是类比。这是架构。

## 人类的新角色：不是用户，是 Spec Sovereign

当 Agent 的概念被移除之后，人类和系统之间的关系也根本性地改变了。

你不再是"和 AI 对话的用户"。你是：

**1. 方向定义者** — 你决定 Hermit 要解决什么问题
**2. 规则制定者** — 你定义什么可以自动执行、什么需要审批、什么风险是不可接受的
**3. 最终仲裁者** — 当多个方案冲突时，你做最终决策
**4. 系统演化的训练师** — 你不是替 Hermit 干活，你在持续提升它的判断体系

你给 Hermit 的不是一条条具体指令。你给的是：

- **Program 级目标**："把 benchmark routing 升级成标准化 profile 体系"
- **治理宪法**：什么风险必须审批、什么证据才算完成
- **验收标准**：测试覆盖率 > 80%，benchmark 不退化
- **风险预算**：夜间可以自动执行低风险任务

Hermit 负责把这些展开成数十个 Task、上百个 Step、经过治理流水线执行，产出 receipts 和 proofs。

**你投的是 Program，不是每一个小 Step。**

这里有一个衡量指标：**Prompt Leverage Ratio = 派生出的有效 Attempt 数 / 人类输入 Prompt 数**。如果你发了 1 个 Program 级 prompt，系统展开成 18 个有效 Attempt，12 个验证通过，3 个自动生成 follow-up——这才是 Task OS 该有的杠杆。

## 治理不是摩擦，是承重结构

很多人听到"治理"会觉得这是束缚。更多审批、更多流程、更慢的速度。

但想一下：为什么你信任操作系统？

不是因为操作系统让程序跑得更快。是因为操作系统保证一个程序的崩溃不会拖死其他所有程序。是因为你知道，即使某个进程行为异常，它能造成的伤害是有边界的。

Hermit 的治理也是同样的道理。

在 Hermit 里，每一个有后果的动作都要经过一条完整的治理流水线：

```
ActionRequest → PolicyEvaluation → Decision → Approval（如需）
→ WorkspaceLease → CapabilityGrant → Execution → Receipt → Reconciliation
```

这里面每一步都产出一条 kernel record：

- **Policy** 决定这个动作属于什么风险类别（21 种 ActionClass）
- **Approval** 记录谁批准了、为什么批准（不是简单的 yes/no）
- **CapabilityGrant** 发放的是有范围、有时限的权限
- **WorkspaceLease** 保证同一时刻只有一个任务在修改同一个工作区
- **Receipt** 是 HMAC-SHA256 签名的结构化记录，不是 log line
- **Reconciliation** 闭合认知环——只有被和解的结果才能变成持久知识

这些不是"nice to have"。它们是系统能够在你不盯着的时候安全运行的**前提条件**。

没有治理流水线，"夜间自动执行"就是"夜间失控执行"。
没有 Receipt，"AI 完成了工作"就是"AI 说它完成了工作"。
没有 Reconciliation，"系统在学习"就是"系统在积累幻觉"。

---

## 睡后干活：Task OS 的终极承诺

Hermit 0.3 最有野心的愿景不是"更好的 AI 编程助手"。是：

**你睡着了，系统还在以你定义的规则继续干活。**

这在 Agent 框架里是不可能的。因为 Agent 框架的原语是 session，session 需要你在。

但在 Task OS 里，这是自然的：

- Task Graph 是持久化的——系统断了可以接着跑
- Park/Resume 让审批不阻塞全局——一个任务等审批，其他任务继续
- 风险分级让低风险工作可以自动执行——research、测试、benchmark 不需要你盯着
- 状态投影让你醒来时能快速了解进展——不用翻几百条 log

你第二天醒来看到的应该是：

> 昨晚完成了 6 个低风险节点。2 个节点等待你审批。1 个 benchmark 退化需要你决策。整体 Program 进度 67%。

而不是：

> [等待用户输入]

## 记忆不能靠"记住了"

在大多数 Agent 系统里，记忆是一个便签系统：把之前对话里的信息存下来，下次用的时候捞出来。

这对问答很好。对后果性工作很危险。

因为没有来源的记忆就是**隐性权威**。三周前存下的一个"事实"，可能今天已经过时。但模型会以同样的置信度使用它——因为它不知道这个记忆是不是还有效。

Hermit 把记忆分成三层：

1. **工作记忆** — 短期的，仅限当前任务
2. **可修订的信念** — 中期的，带有置信度和失效机制
3. **持久记忆** — 长期的，**必须引用证据才能晋升**

最后一点是关键：你不能在没有证据的情况下创建一条持久记忆。如果证据被推翻，对应的记忆会被标记、重新评估或删除。

这意味着系统的"知识"不会随时间腐烂。每一条持久记忆都可以追溯到产出它的证据。当那个证据不再成立时，系统知道去质疑而不是盲目信赖。

## 自迭代：内核信任自己吗？

Hermit 0.3 最有意思的能力之一是**自迭代**——Hermit 可以改进 Hermit 自己。

但这不是"AI 自己改自己的代码"这种危险的叙事。

自迭代在 Hermit 里经过完整的治理流水线：

```
Spec → Research → Decompose → Execute（in isolated workspace）
→ Review → Benchmark → Reconcile → Promote（or Reject）
```

- 每一步都是一个 governed Task
- 代码修改发生在隔离的工作区（git worktree）
- 合并前必须通过 benchmark 验证
- 只有 Reconciliation 通过才能被接受

**内核用自己的治理规则来审查自己的修改。**

这个设计的核心信念是：**自主系统的演化不应该通过移除约束来实现，而应该通过让约束足够智能以允许安全的进化来实现。**

每个 receipt 仍然被签名。每条 proof chain 仍然被哈希链接。每次变更仍然被治理。内核不信任任何人——**包括它自己**。

## Hermit 不是什么

说清楚 Hermit **不是**什么，和说清楚它是什么一样重要：

**不是通用聊天助手。** Hermit 对"帮我改个 typo"这种任务来说太重了。它是为长时间运行、需要审批、值得审计的工作设计的。

**不是云端托管服务。** Hermit 的内核账本是你本地的 SQLite 数据库，状态存在 `~/.hermit`。这不是技术限制，这是信任设计。你的执行历史、审批记录、能力授权——全部在你的机器上，你完全控制。

**不是无代价的。** 治理有开销。Task/Step/Attempt/CapabilityGrant/WorkspaceLease/Receipt/Proof——这些概念需要学习。更多的机制意味着更多的认知负担。Hermit 坦诚地承认这些代价，并赌注对于正确的工作类别，这些代价是值得的。

**不是一个完成品。** Hermit 是 alpha 阶段的软件，有一个清晰的论点和可工作的代码。但内核规范还是草案，不是每个运行时表面都完全实现了每个内核语义。粗糙的边缘、breaking changes、文档和现实之间的差距——都存在。

## 核心信念

Hermit 0.3 建立在一个信念之上：

**AI Agent 需要操作系统，不是框架。**

每一个计算平台最终都需要操作系统。大型机在 1960 年代有了。个人电脑在 1980 年代有了。手机在 2000 年代有了。AI Agent 是下一个计算平台——它们也需要操作系统。

不是一个给你积木然后祝你好运的框架。而是一个对每一个特权操作都介入治理、问责和可恢复性的内核。

**能力和可信赖性不在同一条轴上。** 一个更有能力但删了错误文件的 Agent 并不比一个没那么强但会先问你的 Agent 更好。一个执行完美但不留任何可检查记录的 Agent 不是可信赖的——它只是运气好。

模型会变得越来越强，越来越自主。这个趋势不可逆。

问题不是 Agent 是否会变得更自主。它们会的。

问题是，当它们变得更自主时，操作者是否还能回答：

**发生了什么？为什么被允许发生？可以撤回吗？**

这是一个内核问题。它值得一个内核级的回答。
